# ================================================================
# Therapy Journalling API — DynamoDB Schema Design
# ================================================================
#
# Best Practices Applied:
#   - Access-pattern-first design (NoSQL design philosophy)
#   - Single-table design used where multiple entity types share a partition:
#       JournalTable: JOURNAL metadata + ENTRY items co-located on clientId
#       MessageTable: all thread messages co-located on THREAD#{clientId}#{therapistId}
#   - Natural primary keys (userId, mappingId, appointmentId, sessionId, etc.), no artificial
#     prefixes on single-entity tables. Composite/prefixed keys only where semantically
#     justified: THREAD#{clientId}#{therapistId} (composite thread identifier),
#     clientId#emotion (GSI namespace scoping)
#   - Sort keys only where genuinely needed:
#       Range queries: sentAt#messageId (MessageTable messageSk)
#       Multi-item-type partitions: JOURNAL | entry SK
#       Meaningful composite key: therapistId as SK in RelationshipTable
#       Constant SKs (METADATA, PROFILE) have been REMOVED — they add zero value
#   - Sparse GSIs using optional attributes
#   - Composite GSI partition keys for multi-attribute queries without Scan
#   - Atomic counters for totalEntries via UpdateItem ADD expression
#   - TransactWriteItems for multi-table atomic operations
#   - KEYS_ONLY projections on dedup GSIs to minimise RCU cost
#
# Format Reference:
#   TABLE          Table name
#   DESCRIPTION    Purpose and design rationale
#   ATTRIBUTES     name | Type | Role | Description
#   KEYS           Primary key definition (HASH + RANGE)
#   GSI            Global Secondary Index block (repeatable)
#   LSI            Local Secondary Index block (repeatable)
#   ACCESS_PATTERNS  Per-API query strategy (repeatable)
#   ---            Table separator
#
# Role Values:
#   PK      Partition key of base table
#   SK      Sort key of base table
#   GSI_PK  Partition key of a GSI
#   GSI_SK  Sort key of a GSI
#   LSI_SK  Sort key of an LSI (shares base table PK)
#   ATTR    Regular attribute
# ================================================================

TABLE UserTable

DESCRIPTION
  Single table for both CLIENT and THERAPIST accounts. userId is the natural
  partition key, no prefix needed since this is a single-entity-type table with
  no ambiguity. No sort key — each user has exactly one item. Therapist-only
  attributes (licenseNumber, specialization, bio, qualification, yearsOfExperience)
  are absent on CLIENT items, creating naturally sparse indexes.
  GSI_EmailIndex enables O(1) login lookup without a full table scan.
  GSI_UserTypeIndex powers the therapist discovery endpoint by co-locating
  all THERAPIST items on one GSI partition, sorted by years of experience.

ATTRIBUTES
  userId            String    PK      Unique user ID. Prefix cli_ (client) or thr_ (therapist)
  userType          String    GSI_PK  CLIENT or THERAPIST. GSI_UserTypeIndex partition key
  email             String    GSI_PK  User email. GSI_EmailIndex partition key
  passwordHash      String    ATTR    bcrypt hash. Never included in API responses
  name              String    ATTR    Full display name
  createdAt         String    ATTR    ISO 8601 creation timestamp
  updatedAt         String    ATTR    ISO 8601 last-updated timestamp
  licenseNumber     String    ATTR    THERAPIST only. Professional license number
  specialization    String    ATTR    THERAPIST only. Area of specialization
  bio               String    ATTR    THERAPIST only. Professional biography
  qualification     String    ATTR    THERAPIST only. Academic qualification
  yearsOfExperience  Number    GSI_SK  THERAPIST only. Mandatory. Range key on GSI_UserTypeIndex

KEYS
  HASH    userId

GSI
  NAME        GSI_EmailIndex
  HASH        email
  RANGE       userType
  PROJECTION  INCLUDE userId, userType, email, passwordHash, name, createdAt, updatedAt,
                      licenseNumber, specialization, bio, qualification, yearsOfExperience
  DESCRIPTION O(1) login lookup by email without scanning the full table. userType
              as range key allows exact-match filtering in a single Query.
              INCLUDE projection with passwordHash explicitly listed — the hash is intentionally
              present here for login credential verification, not included by accident.
              Explicit projection ensures any future sensitive attribute added to UserTable
              does not automatically appear in this index without a conscious decision.
              Application enforces email uniqueness at write time via this GSI.

GSI
  NAME        GSI_UserTypeIndex
  HASH        userType
  RANGE       yearsOfExperience
  PROJECTION  INCLUDE userId, userType, name, email, createdAt, updatedAt,
                      licenseNumber, specialization, bio, qualification, yearsOfExperience
  DESCRIPTION Co-locates all THERAPIST items on one GSI partition for GET /therapists.
              yearsOfExperience as range key enables O(1) key-condition range queries
              (>= :min, BETWEEN :min AND :max) without a FilterExpression scan overhead.
              Naturally sparse, CLIENT items carry no yearsOfExperience, keeping the
              index lean. yearsOfExperience is mandatory for therapists so every
              THERAPIST item is guaranteed to appear in the index.
              INCLUDE projection explicitly excludes password — the hash is never needed
              for a therapist discovery query and must never appear in list responses.
              FilterExpression applied post-query for name/specialization/bio/qualification search.
              For production we should use Amazon OpenSearch Service for full-text search.

ACCESS_PATTERNS
  API         POST /auth/clients/register
  OPERATION   Query + PutItem
  INDEX       GSI_EmailIndex then UserTable
  KEYS        userId = :userId
  NOTES       1. Query GSI_EmailIndex (email = :email) to verify email uniqueness — 409 if found.
              2. PutItem with ConditionExpression attribute_not_exists(userId).
              Returns ClientResponse (no passwordHash).

ACCESS_PATTERNS
  API         POST /auth/therapists/register
  OPERATION   Query + PutItem
  INDEX       GSI_EmailIndex then UserTable
  KEYS        userId = :userId
  NOTES       Same flow as client registration with therapist-specific attributes.
              yearsOfExperience is mandatory — it is the sort key on GSI_UserTypeIndex.
              ConditionExpression attribute_not_exists(userId) prevents duplicate IDs.
              Returns TherapistResponse (no passwordHash).

ACCESS_PATTERNS
  API         POST /auth/login
  OPERATION   Query
  INDEX       GSI_EmailIndex
  KEYS        email = :email
  NOTES       Query returns at most one item (one account per email enforced at write time).
              BCrypt.checkpw() validates password against stored hash.
              Returns 401 with the same message for unknown email and wrong password —
              deliberate vagueness prevents user enumeration.
              Returns AuthResponse: token (JWT HS256), tokenType, expiresIn (3600s), userId, userType.

ACCESS_PATTERNS
  API         GET /clients/{clientId}
  OPERATION   GetItem
  INDEX       UserTable
  KEYS        userId = :clientId
  NOTES       O(1) direct lookup. password excluded via ProjectionExpression.
              Therapist callers validated against RelationshipTable before access.

ACCESS_PATTERNS
  API         GET /therapists/{therapistId}
  OPERATION   GetItem
  INDEX       UserTable
  KEYS        userId = :therapistId
  NOTES       O(1) direct lookup. All users can view therapist public profiles.

ACCESS_PATTERNS
  API         GET /therapists
  OPERATION   Query
  INDEX       GSI_UserTypeIndex
  KEYS        userType = THERAPIST
  FILTER      contains(name, :q) OR contains(specialization, :q) OR contains(bio, :q)
  NOTES       yearsOfExperience pushed into KeyConditionExpression as range condition
              (userType = THERAPIST AND yearsOfExperience >= :min) — no scan overhead.
              Omit range condition for unfiltered listing; all therapists present since
              yearsOfExperience is mandatory. FilterExpression for name/specialization/bio
              applied post-query. Paginated with ExclusiveStartKey.

ACCESS_PATTERNS
  API         PUT /clients/{clientId}
  OPERATION   UpdateItem
  INDEX       UserTable
  KEYS        userId = :clientId
  NOTES       UpdateExpression SET supplied fields + updatedAt = :now.
              ConditionExpression attribute_exists(userId) prevents ghost writes.

ACCESS_PATTERNS
  API         PUT /therapists/{therapistId}
  OPERATION   UpdateItem
  INDEX       UserTable
  KEYS        userId = :therapistId
  NOTES       Same pattern as client update. Only therapist-editable fields updated.

ACCESS_PATTERNS
  API         DELETE /clients/{clientId}
  OPERATION   TransactWriteItems (cascade)
  INDEX       UserTable + JournalTable + MappingTable + RelationshipTable
  KEYS        userId = :clientId
  NOTES       TransactWriteItems deletes user item and journal metadata. Cascade
              deletion of entries, mappings, appointments, messages handled
              asynchronously via DynamoDB Streams + Lambda in production.

ACCESS_PATTERNS
  API         DELETE /therapists/{therapistId}
  OPERATION   TransactWriteItems (cascade)
  INDEX       UserTable + SessionTable + MappingTable + RelationshipTable
  KEYS        userId = :therapistId
  NOTES       Cascades to owned sessions, notes, and mappings. Client appointments
              auto-transitioned to CANCELLED via stream processor.

TABLE MappingTable

DESCRIPTION
  Stores formal client-therapist therapeutic relationships. mappingId is the natural partition key, no prefix or sort key needed since this is a single-entity-type table. Three GSIs cover every access pattern: GSI_ClientMappings lists a client's mappings, GSI_TherapistMappings lists a therapist's mappings, and GSI_ClientTherapistLookup enables O(1) conflict detection at mapping creation. Mappings represent a formal long-term therapeutic commitment between the two parties. State machine transitions are enforced via DynamoDB ConditionExpressions to prevent race conditions. TransactWriteItems ensures atomic updates across MappingTable and RelationshipTable.

ATTRIBUTES
  mappingId           String    PK      Unique mapping ID
  clientId            String    GSI_PK  GSI_ClientMappings and GSI_ClientTherapistLookup PK
  therapistId         String    GSI_PK  GSI_TherapistMappings PK. GSI_ClientTherapistLookup SK
  mappingStatus       String    ATTR    PENDING | APPROVED | REJECTED
  journalAccessStatus  String    ATTR    NONE | REQUESTED | GRANTED | REVOKED
  initiatedBy         String    ATTR    CLIENT or THERAPIST. Drives approval logic
  createdAt           String    GSI_SK  ISO 8601. Range key on client and therapist GSIs
  updatedAt           String    ATTR    ISO 8601

KEYS
  HASH    mappingId

GSI
  NAME        GSI_ClientMappings
  HASH        clientId
  RANGE       createdAt
  PROJECTION  ALL
  DESCRIPTION Lists all mappings for a client sorted by creation date.
              Used by GET /mappings (CLIENT caller) and for existence checks.

GSI
  NAME        GSI_TherapistMappings
  HASH        therapistId
  RANGE       createdAt
  PROJECTION  ALL
  DESCRIPTION Lists all mappings for a therapist sorted by creation date.
              Used by GET /mappings (THERAPIST caller).

GSI
  NAME        GSI_ClientTherapistLookup
  HASH        clientId
  RANGE       therapistId
  PROJECTION  KEYS_ONLY
  DESCRIPTION Point lookup for a specific client+therapist pair. Used for:
              1. POST /mappings — conflict detection (409 if mapping exists).
              KEYS_ONLY projection sufficient — only existence matters, mapping
              data is never needed from this index.

ACCESS_PATTERNS
  API         POST /mappings
  OPERATION   Query + PutItem
  INDEX       GSI_ClientTherapistLookup then MappingTable
  KEYS        clientId = :clientId, therapistId = :therapistId
  NOTES       1. Query GSI_ClientTherapistLookup — return 409 if any item returned.
              2. PutItem with ConditionExpression attribute_not_exists(mappingId).
              3. Set journalAccessStatus = REQUESTED (therapist) or GRANTED (client)
                 if journalAccess = true in the request body.
              4. UpdateItem RelationshipTable to add MAPPING to relationshipTypes.

ACCESS_PATTERNS
  API         GET /mappings
  OPERATION   Query
  INDEX       GSI_ClientMappings (CLIENT) | GSI_TherapistMappings (THERAPIST)
  KEYS        clientId = :callerId | therapistId = :callerId
  FILTER      mappingStatus = :status (optional), journalAccessStatus = :jas (optional)
  NOTES       Caller type from JWT selects appropriate GSI. Paginated.

ACCESS_PATTERNS
  API         GET /mappings/{mappingId}
  OPERATION   GetItem
  INDEX       MappingTable
  KEYS        mappingId = :mappingId
  NOTES       O(1) direct lookup. Application verifies clientId or therapistId
              matches JWT caller — returns 403 otherwise (not 404).

ACCESS_PATTERNS
  API         PATCH /mappings/{mappingId}/mapping-status
  OPERATION   UpdateItem
  INDEX       MappingTable
  KEYS        mappingId = :mappingId
  NOTES       ConditionExpression: mappingStatus = PENDING AND initiatedBy <> :callerType.
              Rejects if caller is the initiating party (403) or wrong transition (422).
              On APPROVED: UpdateItem RelationshipTable to sync mappingStatus.

ACCESS_PATTERNS
  API         PATCH /mappings/{mappingId}/journal-access
  OPERATION   TransactWriteItems
  INDEX       MappingTable + RelationshipTable
  KEYS        mappingId = :mappingId
  NOTES       ConditionExpression enforces per-caller allowed transitions.
              Auto-approve: if client sets GRANTED and mappingStatus = PENDING,
              TransactWriteItems atomically sets journalAccessStatus = GRANTED
              AND mappingStatus = APPROVED in one call. RelationshipTable synced.

ACCESS_PATTERNS
  API         DELETE /mappings/{mappingId}
  OPERATION   TransactWriteItems
  INDEX       MappingTable + RelationshipTable
  KEYS        mappingId = :mappingId
  NOTES       Atomically: 1. DeleteItem mapping.
              2. DELETE MAPPING from RelationshipTable relationshipTypes StringSet.
              If set becomes empty, delete the RelationshipTable item entirely.

TABLE RelationshipTable

DESCRIPTION
  Denormalised aggregation table tracking ALL connections between a client and
  therapist regardless of type (MAPPING, APPOINTMENT, MESSAGE). Written through
  whenever any connection is created or removed. Eliminates cross-table joins
  for the connected-parties list endpoints. Eventual consistency with source
  tables is acceptable for list views; authoritative checks (e.g. APPROVED
  mapping for booking) always query source tables directly. The StringSet
  relationshipTypes supports atomic ADD/DELETE operations — idempotent and
  safe for concurrent writes.

ATTRIBUTES
  clientId            String     PK      Client user ID. Natural partition key
  therapistId         String     SK      Therapist user ID. Natural sort key — uniquely identifies the client-therapist pair. Meaningful range key: enables querying all therapists for a client (base table) or all clients for a therapist (GSI_TherapistRelationships)
  relationshipTypes   StringSet  ATTR    Active types: {MAPPING, APPOINTMENT, MESSAGE}
  mappingId           String     ATTR    Present if a mapping exists. Absent otherwise
  mappingStatus       String     ATTR    Current mapping status. Absent if no mapping
  journalAccessStatus  String     ATTR    Current journal access status. Absent if no mapping
  createdAt           String     ATTR    Timestamp of first relationship
  updatedAt           String     ATTR    Last modification timestamp

KEYS
  HASH    clientId
  RANGE   therapistId

GSI
  NAME        GSI_TherapistRelationships
  HASH        therapistId
  RANGE       clientId
  PROJECTION  ALL
  DESCRIPTION Therapist-keyed view of all connected clients.
              Used by GET /therapists/{therapistId}/clients. Mirrors the base
              table query pattern from the client perspective.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/therapists
  OPERATION   Query
  INDEX       RelationshipTable
  KEYS        clientId = :clientId
  FILTER      contains(relationshipTypes, :type) (optional)
  NOTES       Returns all relationship items for this client. Full therapist profiles
              fetched via BatchGetItem on UserTable using therapistId values. Paginated.

ACCESS_PATTERNS
  API         GET /therapists/{therapistId}/clients
  OPERATION   Query
  INDEX       GSI_TherapistRelationships
  KEYS        therapistId = :therapistId
  FILTER      contains(relationshipTypes, :type) (optional)
  NOTES       Full client profiles fetched via BatchGetItem on UserTable. Paginated.

TABLE JournalTable

DESCRIPTION
  Single-table design for journal metadata and emotion entries. Both item types
  share partition key clientId, co-locating a client's metadata and all their
  entries on the same DynamoDB partition for efficient reads. Two item types:
  JOURNAL (one per client, metadata only) and entries identified by a stable
  journalEntryId sort key. journalEntryId is a server-generated UUID, immutable,
  requires no client-side key tracking, and enables O(1) single-item operations
  without any auxiliary key. emotionTimestamp is a plain mutable attribute:
  updating it is a simple UpdateItem with no sort key migration.
  LSI_EmotionTime provides time-ordered listing by emotionTimestamp on the same
  partition (no extra RCU hop). LSI_IntensityTime handles intensity+date queries
  on the same partition. GSI_EmotionEntries uses composite key clientId#emotion
  to scope the GSI partition to one client+emotion, preventing cross-client leakage.
  totalEntries on the JOURNAL item is maintained as an atomic counter.
  Access control: therapist may read only if mappingStatus=APPROVED and
  journalAccessStatus=GRANTED.

ATTRIBUTES
  clientId          String    PK      Client user ID. Natural partition key (all item types)
  journalEntryId    String    SK      JOURNAL (metadata item) | <uuid> (entry items)
  totalEntries      Number    ATTR    Atomic entry count. Present on JOURNAL item only
  emotion           String    ATTR    Happiness | Caring | Depression | Inadequate | Fear | Confusion | Hurt | Anger | Loneliness | Remorse
  feeling           String    ATTR    Specific feeling within the emotion category
  intensity         String    ATTR    STRONG | MEDIUM | LIGHT
  emotionTimestamp  String    LSI_SK  When the emotion was actually felt (ISO 8601). LSI_EmotionTime range key
  intensityTime     String    LSI_SK  Composite: intensity#emotionTimestamp. LSI_IntensityTime range key
  clientEmotion     String    GSI_PK  Composite: clientId#emotion. GSI_EmotionEntries PK
  notes             String    ATTR    Optional free-text from client
  createdAt         String    ATTR    ISO 8601 creation timestamp
  updatedAt         String    ATTR    ISO 8601 last-updated timestamp

KEYS
  HASH    clientId
  RANGE   journalEntryId

LSI
  NAME        LSI_EmotionTime
  RANGE       emotionTimestamp
  PROJECTION  ALL
  DESCRIPTION Time-ordered entry listing per client by when the emotion was felt.
              Query: clientId = :cid, emotionTimestamp BETWEEN :from AND :to.
              ScanIndexForward=true for chronological order, false for reverse.
              LSI shares the base table PK — no extra partition hop, lower latency and cost.
              Naturally sparse — JOURNAL metadata item carries no emotionTimestamp attribute.

LSI
  NAME        LSI_IntensityTime
  RANGE       intensityTime
  PROJECTION  ALL
  DESCRIPTION Intensity+date range queries within a single client's journal.
              intensityTime = intensity#emotionTimestamp (e.g. STRONG#2024-06-01T10:00:00Z).
              BETWEEN queries cover a specific intensity bucket over a date range.
              LSI shares the base table PK — no extra partition hop, lower latency and cost.

GSI
  NAME        GSI_EmotionEntries
  HASH        clientEmotion
  RANGE       emotionTimestamp
  PROJECTION  ALL
  DESCRIPTION Emotion-category filter for a specific client's journal entries.
              clientEmotion composite key (clientId#emotion) scopes the GSI partition
              to one client+emotion combination, preventing cross-client data leakage
              and keeping individual GSI partitions small. emotionTimestamp range key
              enables optional date range filtering in the same Query call.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/journal
  OPERATION   GetItem
  INDEX       JournalTable
  KEYS        clientId = :clientId, journalEntryId = JOURNAL
  NOTES       O(1) metadata lookup. Returns totalEntries counter and journal metadata.
              Journal is owned by the client and identified by clientId alone — no separate journalId.
              Therapist callers must have mappingStatus=APPROVED + journalAccessStatus=GRANTED.

ACCESS_PATTERNS
  API         POST /clients/{clientId}/journal/entries
  OPERATION   TransactWriteItems
  INDEX       JournalTable
  KEYS        clientId = :clientId, journalEntryId = <server-generated uuid>
  NOTES       Atomically: 1. PutItem new entry item (journalEntryId = server-generated uuid).
              2. UpdateItem JOURNAL item: ADD totalEntries 1, SET updatedAt = :now.
              Compute clientEmotion = clientId#emotion and
              intensityTime = intensity#emotionTimestamp before writing.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/journal/entries (date range)
  OPERATION   Query
  INDEX       LSI_EmotionTime
  KEYS        clientId = :clientId, emotionTimestamp BETWEEN :from AND :to
  NOTES       Time-ordered by emotionTimestamp via LSI — no GSI required.
              ScanIndexForward=true for chronological order.
              Paginated with ExclusiveStartKey. intensity FilterExpression combinable.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/journal/entries (emotion filter)
  OPERATION   Query
  INDEX       GSI_EmotionEntries
  KEYS        clientEmotion = {clientId}#{emotion}, emotionTimestamp BETWEEN :from AND :to
  NOTES       Fully scoped to one client+emotion. Optional date range on GSI sort key.
              intensity FilterExpression combinable for further narrowing.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/journal/entries (intensity filter)
  OPERATION   Query
  INDEX       LSI_IntensityTime
  KEYS        clientId = :clientId, intensityTime BETWEEN {intensity}#{from} AND {intensity}#{to}
  NOTES       LSI query — no extra partition hop. Composite sort key handles intensity
              and date in one expression with zero FilterExpression overhead.

ACCESS_PATTERNS
  API         GET /clients/{clientId}/journal/entries/{journalEntryId}
  OPERATION   GetItem
  INDEX       JournalTable
  KEYS        clientId = :clientId, journalEntryId = :journalEntryId
  NOTES       O(1) direct lookup by journalEntryId. No emotionTimestamp required.

ACCESS_PATTERNS
  API         PUT /clients/{clientId}/journal/entries/{journalEntryId}
  OPERATION   UpdateItem
  INDEX       JournalTable
  KEYS        clientId = :clientId, journalEntryId = :journalEntryId
  NOTES       Simple UpdateItem regardless of whether emotionTimestamp changes —
              it is a plain attribute, no sort key migration needed.
              Re-compute and SET intensityTime and clientEmotion if emotion,
              intensity, or emotionTimestamp is updated.

ACCESS_PATTERNS
  API         DELETE /clients/{clientId}/journal/entries/{journalEntryId}
  OPERATION   TransactWriteItems
  INDEX       JournalTable
  KEYS        clientId = :clientId, journalEntryId = :journalEntryId
  NOTES       Atomically: 1. DeleteItem entry item.
              2. UpdateItem JOURNAL item: ADD totalEntries -1, SET updatedAt = :now.

TABLE MessageTable

DESCRIPTION
  Thread-keyed table for client-therapist messages. threadKey
  (THREAD#{clientId}#{therapistId}) co-locates all messages for a conversation
  on the same partition for efficient sequential reads. Sort key messageSk (sentAt#messageId) provides natural chronological ordering and guarantees uniqueness for simultaneous messages.
  GSI_MessageById enables O(1) single-message lookup by messageId.
  senderId and senderType are derived server-side from the JWT, never from
  the request body, to prevent spoofing.

ATTRIBUTES
  threadKey    String    PK      Format: THREAD#{clientId}#{therapistId}. Composite — no single attribute uniquely identifies a thread
  messageSk    String    SK      Format: sentAt#messageId. Chronological order with UUID suffix for uniqueness
  messageId    String    GSI_PK  Unique message ID. GSI_MessageById partition key
  clientId     String    ATTR    Client in this conversation thread
  therapistId  String    ATTR    Therapist in this conversation thread
  senderType   String    ATTR    CLIENT or THERAPIST. Set from JWT, never from request body
  senderId     String    ATTR    userId of sender. Set from JWT, never from request body
  content      String    ATTR    Message body. Max 2000 characters
  sentAt       String    ATTR    ISO 8601 send timestamp
  editedAt     String    ATTR    ISO 8601 edit timestamp. Present only if the message has been edited

KEYS
  HASH    threadKey
  RANGE   messageSk

GSI
  NAME        GSI_MessageById
  HASH        messageId
  RANGE       sentAt
  PROJECTION  ALL
  DESCRIPTION O(1) single-message lookup by messageId. Only message items carry
              messageId so no unrelated items pollute this index partition.
              Used by PUT /messages/{messageId} and DELETE /messages/{messageId}.

ACCESS_PATTERNS
  API         POST /messages
  OPERATION   PutItem
  INDEX       MessageTable
  KEYS        threadKey = THREAD#{clientId}#{therapistId}, messageSk = sentAt#messageId
  NOTES       senderId and senderType written from JWT — request body values ignored.
              UpdateItem RelationshipTable: ADD MESSAGE to relationshipTypes StringSet.
              ADD is idempotent — safe even if MESSAGE already present in the set.

ACCESS_PATTERNS
  API         GET /messages?clientId=X&therapistId=Y
  OPERATION   Query
  INDEX       MessageTable
  KEYS        threadKey = THREAD#{clientId}#{therapistId}
  NOTES       KeyConditionExpression on threadKey alone — messageSk format (sentAt#messageId)
              ensures natural chronological ordering without a range condition.
              Oldest-first: ScanIndexForward = true (default). Newest-first: ScanIndexForward = false.
              Caller validated as clientId or therapistId via JWT before query.
              Paginated with ExclusiveStartKey on messageSk.

ACCESS_PATTERNS
  API         PUT /messages/{messageId}
  OPERATION   Query then UpdateItem
  INDEX       GSI_MessageById then MessageTable
  KEYS        messageId = :messageId (to retrieve threadKey and messageSk), then threadKey + messageSk to update
  NOTES       1. Query GSI_MessageById to retrieve threadKey, messageSk, senderId, and sentAt.
              2. Validate caller matches senderId (only sender may edit).
              3. Validate sentAt is within 10 minutes of now (edit window).
              4. UpdateItem on base table: SET content = :c, editedAt = :ea.

ACCESS_PATTERNS
  API         DELETE /messages/{messageId}
  OPERATION   Query then DeleteItem
  INDEX       GSI_MessageById then MessageTable
  KEYS        messageId = :messageId (to retrieve threadKey and messageSk), then threadKey + messageSk to delete
  NOTES       1. Query GSI_MessageById to retrieve threadKey and messageSk values.
              2. Validate caller matches senderId (only sender can delete own message).
              3. DeleteItem on base table using retrieved threadKey and messageSk.

TABLE SessionTable

DESCRIPTION
  Single item per session. Session metadata, private notes, shared notes, and
  confirmed appointment reference are all stored on one item keyed by sessionId.
  No sort key, each session has exactly one item, eliminating item-type management.
  privateNotes and sharedNotes are plain attributes with enforced size limits;
  access control is application-layer (privateNotes never serialised for CLIENT callers).
  confirmedAppointmentId and pendingCount are denormalised from AppointmentTable so the
  UI can render session + appointment state in a single read without cross-table joins.
  GSI projections use INCLUDE (not ALL) to exclude note attributes from list queries,
  note content is only fetched on single-item reads, keeping list RCU cost low.
  GSI_TherapistSessions lists a therapist's sessions sorted by scheduled time.
  GSI_StatusSchedule powers the client-facing session browser.

ATTRIBUTES
  sessionId               String    PK      Unique session ID. Natural partition key
  therapistId             String    GSI_PK  GSI_TherapistSessions PK
  title                   String    ATTR    Session title. Max 200 characters
  description             String    ATTR    Session description. Max 1000 characters
  scheduledAt             String    GSI_SK  ISO 8601. Range key on both GSIs
  durationMinutes         Number    ATTR    Duration in minutes. Minimum 15
  isAvailable             Boolean   ATTR    Whether slot accepts new appointment requests
  status                  String    GSI_PK  SCHEDULED|IN_PROGRESS|COMPLETED|CANCELLED. GSI_StatusSchedule PK
  startedAt               String    ATTR    ISO 8601. Absent until session started
  endedAt                 String    ATTR    ISO 8601. Absent until session ended
  confirmedAppointmentId  String    ATTR    ID of the CONFIRMED appointment. Absent if unbooked. Cleared on cancellation
  pendingCount            Number    ATTR    Count of PENDING appointment requests. Maintained as atomic counter
  privateNotes            String    ATTR    Therapist-only notes. Max 10 000 characters. Never serialised for CLIENT callers
  sharedNotes             String    ATTR    Client-visible session notes. Max 5 000 characters
  createdAt               String    ATTR    ISO 8601 creation timestamp
  updatedAt               String    ATTR    ISO 8601 last-updated timestamp

KEYS
  HASH    sessionId

GSI
  NAME        GSI_TherapistSessions
  HASH        therapistId
  RANGE       scheduledAt
  PROJECTION  INCLUDE sessionId, therapistId, title, scheduledAt, durationMinutes,
                      isAvailable, status, confirmedAppointmentId, pendingCount,
                      createdAt, updatedAt

TABLE AppointmentTable

DESCRIPTION
  Stores appointment requests from clients for therapist session slots
  Four GSIs cover every access pattern. GSI_ClientAppointments and GSI_TherapistAppointments support paginated list views. GSI_SessionAppointments groups appointments by session and status — critical for confirming one and batch-rejecting others. GSI_ClientSessionDedup uses composite key clientId#sessionId for O(1) duplicate prevention with KEYS_ONLY projection to minimise read cost. The CONFIRM transition is the most complex write: sequential updates confirm the target, reject all other PENDING appointments for the same session, and flip isAvailable=false on SessionTable.

ATTRIBUTES
  appointmentId   String    PK      Unique appointment ID. Natural partition key. No prefix needed — single entity type
  sessionId       String    GSI_PK  GSI_SessionAppointments partition key
  clientId        String    GSI_PK  GSI_ClientAppointments partition key
  therapistId     String    GSI_PK  GSI_TherapistAppointments partition key
  status          String    GSI_SK  PENDING|CONFIRMED|CANCELLED|COMPLETED|REJECTED
  clientSession   String    GSI_PK  Composite: clientId#sessionId. Dedup GSI PK
  requestedAt     String    GSI_SK  ISO 8601. Range key on client and therapist GSIs
  updatedAt       String    ATTR    ISO 8601

KEYS
  HASH    appointmentId

GSI
  NAME        GSI_ClientAppointments
  HASH        clientId
  RANGE       requestedAt
  PROJECTION  ALL
  DESCRIPTION Lists all appointments for a client sorted by request time.
              Used by GET /appointments (CLIENT caller).

GSI
  NAME        GSI_TherapistAppointments
  HASH        therapistId
  RANGE       requestedAt
  PROJECTION  ALL
  DESCRIPTION Lists all appointments across all sessions for a therapist.
              Used by GET /appointments (THERAPIST caller).

GSI
  NAME        GSI_SessionAppointments
  HASH        sessionId
  RANGE       status
  PROJECTION  ALL
  DESCRIPTION Groups appointments by session and status. Used for:
              1. POST /sessions/{id}/start — verify CONFIRMED appointment exists.
              2. PATCH CONFIRM — find all PENDING to batch-reject.

GSI
  NAME        GSI_ClientSessionDedup
  HASH        clientSession
  RANGE       status
  PROJECTION  KEYS_ONLY
  DESCRIPTION Composite key clientId#sessionId scopes dedup to one client+session.
              KEYS_ONLY projection minimises RCU cost — only existence matters.
              Used by POST /appointments to enforce one-per-client-per-session rule.

ACCESS_PATTERNS
  API         POST /appointments
  OPERATION   Query (x2 checks) then PutItem + UpdateItem
  INDEX       GSI_ClientSessionDedup + SessionTable + RelationshipTable
  KEYS        clientSession = {clientId}#{sessionId}
  NOTES       1. Query GSI_ClientSessionDedup — return 409 if item found.
              2. GetItem SessionTable (sessionId = :sessionId) — return 422 if isAvailable=false.
              3. PutItem appointment with status=PENDING, requestedAt=now.
              4. UpdateItem RelationshipTable: ADD APPOINTMENT to relationshipTypes (upsert).
                 No mapping is required — clients may book one-off appointments with any
                 therapist who has an available session, independent of any mapping.
